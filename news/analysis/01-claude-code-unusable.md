# 深度解读：Claude Code 在复杂工程任务中不可用

> 原文：[GitHub Issue #42796](https://github.com/anthropics/claude-code/issues/42796)
> HN 热度：⬆️ 1307 pts / 💬 720 评论
> 发布日期：2026-04-06

---

## 核心论点

一位使用 Claude Code 做 C/GPU 驱动开发的工程师，用 **17,871 个思考块 + 234,760 次工具调用** 的数据证明了：**Anthropic 在 2026 年 2-3 月间悄悄削减了 Claude 的深度思考能力，直接导致工程质量大幅退化。**

这不是"AI 变笨了"的主观感受——而是有硬数据的系统性行为退化。

---

## 关键时间线

| 时间段 | 思考可见 | 思考脱敏 | 用户感知 |
|--------|---------|---------|---------|
| 1月30日 - 3月4日 | 100% | 0% | 表现优秀 |
| 3月5日 | 98.5% | 1.5% | 未察觉 |
| 3月7日 | 75.3% | 24.7% | 未察觉 |
| 3月8日 | 41.6% | 58.4% | **退化被发现** |
| 3月10-11日 | <1% | >99% | 社区大规模反馈 |
| 3月12日+ | 0% | 100% | 完全不可见 |

质量退化在 3 月 8 日被独立报告——**恰好是脱敏思考块越过 50% 的那一天**。这不是巧合。

## 数据铁证

### 1. 思考深度早已在脱敏前下降

通过 thinking signature 字段（与思考内容长度 Pearson 相关系数 0.971）反推：

| 时期 | 估算中位思考长度 | vs 基线 |
|------|-----------------|---------|
| 1月30日 - 2月8日（基线） | ~2,200 字符 | — |
| 2月下旬 | ~720 字符 | **-67%** |
| 3月1-5日 | ~560 字符 | **-75%** |
| 3月12日+（完全脱敏） | ~600 字符 | **-73%** |

思考深度在 2 月就下降了 67%，脱敏只是让这个过程变得不可观测。

### 2. Read:Edit 比例暴跌

| 时期 | Read:Edit | 研究:变更 | 读占比 | 写占比 |
|------|-----------|----------|--------|--------|
| 良好期（1/30 - 2/12） | **6.6** | 8.7 | 46.5% | 7.1% |
| 过渡期（2/13 - 3/7） | 2.8 | 4.1 | 37.7% | 13.2% |
| 退化期（3/8 - 3/23） | **2.0** | 2.8 | 31.0% | 15.4% |

模型从"每编辑一个文件前读 6.6 个相关文件"变成"只读 2 个就直接改"——**70% 的研究行为消失了**。

### 3. "最简修复"心理

"simplest" 一词的使用频率从 2.7/千次工具调用上升到 6.3/千次——模型倾向偷懒而非做正确的事。

### 4. 推诿/偷懒行为统计

通过编程式 stop hook 捕获的行为：

| 类别 | 3/8-3/25 次数 | 示例 |
|------|-------------|------|
| 推卸责任 | 73 | "不是我改坏的"、"本来就有问题" |
| 请求许可 | 40 | "要我继续吗？"、"要继续往下走吗？" |
| 过早停止 | 18 | "这是个好停顿点"、"自然检查点" |
| 已知限制标签 | 14 | "已知限制"、"以后再说" |
| 会话长度借口 | 4 | "新开会话继续吧"、"太长了" |
| **总计** | **173** | **此前为零次** |

### 5. 用户打断率暴增

用户打断（Escape 键 / 手动纠正）率增加了 **12 倍**——从 0.9/千次工具调用到 11.4/千次。

## 为什么深度思考是结构性的？

受影响的工作流特征：
- **50+ 并发 Agent 会话**，做系统编程（C, MLIR, GPU 驱动）
- **30+ 分钟自主运行**，涉及多文件变更
- **5000+ 字的 CLAUDE.md** 项目约定
- 代码审查、工单管理、迭代调试

深度思考是模型执行以下能力的机制：
1. **多步规划**：决定先读哪些文件、什么顺序
2. **约定遵循**：从 CLAUDE.md 中回忆和应用项目规则
3. **自我纠错**：在输出前捕获自己的错误
4. **会话管理**：判断任务是否完成、是否继续工作
5. **跨工具推理**：在数百次工具调用中保持连贯推理

当思考变浅，模型默认选择最省力的行为：**不读就改、不干就停、推卸责任、选最简单的方案**。

---

## 行为退化详细目录

### 盲改文件
- 良好期：6.2% 的编辑发生在未读文件上
- 退化期：**33.7%** 的编辑发生在未读文件上
- 后果：代码破坏、约定违反、注释块被截断、逻辑重复

### 推理循环
- 良好期：8.2 次/千次工具调用
- 退化期：26.6 次/千次工具调用（**3 倍**）
- 表现："哦等等"、"实际上"、"让我重新考虑"——模型在输出中自我矛盾

### 全文覆写 vs 精准编辑
- 良好期：4.9% 的变更使用全文 Write
- 退化期：**11.1%** — 翻倍。模型越来越倾向于重写整个文件

---

## 作者的建议

1. **透明化思考分配**：如果思考 token 被削减或限制，重度用户需要知道
2. **提供"最大思考"付费档**：需要 20,000 思考 token 的用户愿意为 200 的用户付更多钱
3. **API 响应中暴露 thinking_tokens**：即使内容脱敏，让用户监控推理深度
4. **金丝雀指标**：stop hook 违规率（0 → 10/天）可以作为质量回退的前导指标

## 研究方法论

- **数据来源**：6,852 个 Claude Code 会话 JSONL 文件
- **思考块**：17,871 个（7,146 有内容，10,725 被脱敏）
- **工具调用**：234,760 次
- **用户提示**：18,000+ 条
- **日期范围**：2026 年 1 月 30 日 – 4 月 1 日

## 启示

这是 **AI 工程化落地路上的一记警钟**。当提供商为了降低成本而削减推理预算时，终端用户的体验会"静默退化"——尤其对重度用户，影响是毁灭性的。Claude Code 的退化不是 bug，而是 **成本优化的副产品**，只是 Anthropic 选择不告诉用户。

对于所有依赖 AI 编程工具的团队，这个案例说明：**你需要建立自己的质量监控系统，不能盲目信任提供商的"持续改进"叙事。**
