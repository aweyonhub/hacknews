## 🎯 核心内容/论点

这篇文章深刻剖析了 **Windows 平台原生应用开发的碎片化困境**。作者指出，微软在过去十年中推出了大量互相冲突的应用框架，让 Windows 开发者陷入选择困难。

### 🤯 框架大杂烩

当前 Windows 开发者面临的选择迷宫：

| 框架 | 状态 | 适用场景 | 问题 |
|------|------|----------|------|
| **Win32 / WinAPI** | ✅ 经典稳定 | 传统桌面应用 | 老旧但可靠 |
| **WPF** | ✅ 维护模式 | 企业桌面应用 | .NET 生态绑定 |
| **WinForms** | 📍 遗留支持 | 快速原型 | 已过时 |
| **UWP / WinRT** | ❌ 事实上已死 | Windows Store 应用 | 微软已放弃 |
| **WinUI 2.0** | ⚠️ 仅限 UWP | UWP 样式控件 | 受限于 UWP |
| **WinUI 3.0** | ⚠️ 不稳定 | "现代" Windows 应用 | Bug 多、文档差 |
| **WinAppSDK** | 🔄 发展中 | 解耦 WinUI 运行时 | 依赖链复杂 |
| **MAUI** | ⚠️ 争议巨大 | 跨平台（含 Windows）| 质量问题频出 |
| **PWA / WebView2** | 📍 补充方案 | 轻量混合应用 | 本质是 Web |

### 💥 核心痛点

1. **方向反复摇摆**：从 UWP → WinUI → MAUI → WinAppSDK，每 2 年换一次"官方推荐"
2. **文档质量参差不齐**：WinUI 3 文档被社区广泛吐槽
3. **API 表面不一致**：不同框架之间概念冲突（如导航模式、生命周期）
4. **第三方框架更靠谱**：Avalonia、Uno Platform 等社区方案反而更成熟

## 💡 关键洞察 / 为什么重要

### 🏗️ **微软的平台战略困境**

Windows 应用开发困境的本质是**微软身份认同危机**：

```
想要成为：现代跨平台开发目标 ←→ 但必须维护：30 年 Win32 向后兼容
```

这导致：
- 新框架永远无法彻底取代旧框架
- 旧框架又不能直接废弃（太多依赖）
- 结果就是**所有框架都半死不活**

### 🎯 **社区共识：回归 Win32**

HN 讨论中最强烈的声音是：

> **"stick with Win32" — 这是你可以轻易避免的一团糟**

理由很务实：
- Win32 API 经过 30+ 年验证，稳定可靠
- 大量现有代码和知识积累
- 新框架承诺的美好从未兑现

### 📊 **对行业的启示**

这个案例说明了一个普遍规律：
> **当一家公司无法果断砍掉 legacy 时，所有新尝试都会被拖累**

对比 Apple（敢于砍掉 32 位、Intel）、Google（敢于 kill 产品），微软的"兼容性包袱"成了创新的枷锁。
