## 🎯 核心内容/论点

这是一篇对 **Anthropic Claude Code 在项目中生成的 `.claude/` 配置目录**的完整解剖指南。随着 Claude Code 成为越来越流行的 AI 编程助手，`.claude/` 文件夹正在成为新的 **dotfiles 文化核心**——类似于 `.vimrc`、`.gitconfig` 之于它们各自的生态。

### 📁 .claude/ 目录结构

```
your-project/
└── .claude/
    ├── settings.json          # 项目级设置
    ├── mcp.json               # MCP 服务器配置
    ├── memory/                # 持久化记忆存储
    │   ├── global.md          # 全局记忆
    │   └── project-specific.md
    ├── CLAUDE.md              # 项目指令文件（类似 .github/CO.md）
    ├── allowed-tools.json     # 工具权限白名单
    └── (可选) custom-commands/ # 自定义斜杠命令
```

### 📄 各文件详解

#### 1️⃣ `settings.json` — 核心配置

```json
{
  "model": "claude-sonnet-4-20250514",
  "small_model": "claude-haiku-3-5-20241022",
  "permissions": {
    "allow": ["Read", "Write", "Edit", "Bash(safe commands)"],
    "deny": ["Bash(rm -rf *)", "Network(external)"]
  },
  "autoAccept": {
    "mode": "permissive",      // strict | permissive | off
    "tools": ["Read", "Edit"]  // 自动接受的操作类型
  },
  "max_turns": 100,
  "thinking": {
    "budget_tokens": 10000,
    "enabled": true
  }
}
```

**关键设置解读：**
- **model vs small_model**：主模型用于复杂推理，轻模型用于简单任务（如生成标题），省钱
- **autoAccept mode**：`permissive` 模式下只读/编辑操作自动确认，写/网络操作仍需手动确认
- **thinking budget**：扩展思维的 token 预算，适合复杂推理任务

#### 2️⃣ `mcp.json` — MCP 服务器配置

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["@anthropic/mcp-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "uvx",
      "args": ["mcp-server-postgres", "dburl=postgresql://..."]
    },
    "filesystem": {
      "command": "npx",
      "args": ["@modelcontextprotocol/server-filesystem", "/path/to/allowed/dir"]
    },
    "linear": {
      "command": "npx",
      "args": ["@anthropic/mcp-linear"],
      "env": { "LINEAR_API_KEY": "${LINEAR_KEY}" }
    }
  }
}
```

**MCP（Model Context Protocol）让 Claude Code 能够：**
- 直接查询 GitHub Issues/PR
- 执行 SQL 查询数据库
- 操作指定目录下的文件系统
- 管理 Linear 工单
- 以及更多...

#### 3️⃣ `memory/` — 记忆系统

```
.claude/memory/
├── global.md           # 跨所有项目的持久记忆
├── conventions.md      # 团队编码规范
├── architecture.md    # 系统架构概要
└── context.md          # 项目背景信息
```

**示例 memory/global.md：**
```markdown
# Project Memory

## Team Conventions
- We use TypeScript with strict mode
- All API routes follow RESTful conventions
- Tests use Vitest with coverage >80%

## Architecture
- Frontend: Next.js 14 + React 18
- Backend: FastAPI + PostgreSQL
- Infra: AWS (Terraform managed)

## Common Pitfalls
- Don't use `any` type without justification
- Database migrations must be reversible
- Environment variables are in .env.example only
```

#### 4️⃣ `CLAUDE.md` — 项目指令文件

这是**最重要的文件**——相当于给 AI 的"项目 README"：

```markdown
# Project Instructions

## Overview
This is a SaaS analytics platform for e-commerce businesses.

## Tech Stack
- Language: TypeScript (strict)
- Framework: Next.js 14 (App Router)
- Database: PostgreSQL via Prisma ORM
- Auth: NextAuth.js v5
- Deployment: Vercel

## Code Style
- Use functional components, no class components
- Prefer `const` over `let`
- Follow conventional commits for git messages
- Run `pnpm lint` before committing

## Testing
- Unit tests: Vitest
- E2E: Playwright
- CI runs on every PR

## Things Claude Should Know
- The `lib/` directory contains shared utilities
- Never modify `migrations/` directly — use `prisma migrate dev`
- Environment secrets are in Vercel dashboard, not .env
- The API rate limit is 1000 req/min per user

## What NOT to Do
- Don't add dependencies without team discussion
- Don't modify prisma schema without running migration
- Don't use console.log in production code
```

## 💡 关键洞察 / 为什么重要

### 🔄 **".claude 已经成为新的 dotfiles"**

HN 上的一条高赞评论精准地概括了这个趋势：

> **".claude has become the new dotfiles."**

这意味着：

| 时代 | dotfiles 核心 | 代表文化 |
|------|--------------|----------|
| 1990s | `.cshrc`, `.login` | Unix 系统定制 |
| 2000s | `.vimrc`, `.emacs` | 编辑器定制 |
| 2010s | `.zshrc`, `.tmux.conf` | 终端环境定制 |
| 2020s | `.gitconfig`, `.npmrc` | 开发工具链定制 |
| **2026+** | **`.claude/`** | **AI 编程助手定制** |

### 🎯 **CLAUDE.md 是最重要的投资**

在所有 `.claude/` 文件中，**CLAUDE.md 的 ROI 最高**：

| 投入 | 回报 |
|------|------|
| 花 30 分钟写好 CLAUDE.md | 每次 AI 编程交互都更准确 |
| 明确编码规范 | 减少来回修改的次数 |
| 记录架构决策 | AI 能做出符合项目风格的建议 |
| 列出禁忌事项 | 避免低级错误反复出现 |

**类比**：CLAUDE.md 对 Claude Code 就像 `.editorconfig` + `CONTRIBUTING.md` + `ARCHITECTURE.md` 的综合体。

### ⚠️ **争议与批评**

HN 讨论中也出现了不少质疑声音：

> **"The article overall seems filled with internal inconsistencies. Is this what an LLM generated after some edits?"**

主要批评：
1. **过度工程化**：真的需要这么多配置吗？
2. **vendor lock-in**：深度绑定 Anthropic 生态的风险
3. **安全考虑**：`.claude/` 中可能包含敏感信息（API keys 等）
4. **幻觉风险**：AI 可能"记住"错误的上下文

## 📎 实践建议

### ✅ 最佳实践
1. **将 `.claude/` 加入 `.gitignore` 的例外**（团队共享配置）
2. **使用环境变量引用敏感信息**（不要硬编码 key）
3. **定期更新 `memory/` 内容**（过时记忆比没有更糟糕）
4. **为不同环境准备不同的 `settings.json`**（dev/staging/prod）
5. **版本化管理你的 `.claude/` 模板**

### ❌ 常见误区
1. 把 CLAUDE.md 写成 500 页的文档（AI 也读不完）
2. 在 MCP 配置中硬编码密码
3. 从不更新 memory（导致 AI 给出过时建议）
4. 完全依赖默认设置（错失定制化优势）
5. 忽略 permissions 配置（安全风险）
