# ⚠️ GitHub 挣扎于三个九的可用性 (GitHub appears to be struggling with measly three nines availability)

**日期**: 2026-03-23
**原文链接**: [The Register](https://theregister.com) | [HN](https://news.ycombinator.com/item?id=47487584)
**HN 热度**: ~350 points | ~220+ comments

---

## 🎯 核心内容/论点

The Register 报道称 **GitHub 的服务可用性长期徘徊在"三个九"（99.9%）附近**，这对于承载全球软件开发基础设施的平台来说是一个令人不安的数字。

### 📊 什么是"几个九"？可用性等级对照表

| 可用性级别 | 年度停机时间 | 适用场景 | GitHub 的表现 |
|------------|-------------|----------|---------------|
| **90%** ("一个九") | 36.5 天 | 个人博客 | — |
| **99%** ("两个九") | 3.65 天 | 一般网站 | — |
| **99.9%** ("三个九") | **8.76 小时** | 普通 SaaS | ⚠️ **约在此水平** |
| **99.99%** ("四个九") | **52.6 分钟** | 企业级 SaaS | ❌ 未达到 |
| **99.999%** ("五个九") | **5.26 分钟** | 关键基础设施 | ❌❌ 远未达到 |

### 💥 核心问题

1. **CI/CD 中断影响全球开发流水线**
   - Actions 经常性不可用导致构建失败
   - 部署管道被打断，影响产品发布节奏
   - 团队需要维护备用 CI 方案（额外成本）

2. **API 限流和不稳定**
   - 第三方工具依赖 GitHub API 的可靠性
   - 速率限制变化无常
   - GraphQL API 偶发性故障

3. **单点故障风险**
   - 全球开源生态过度依赖单一平台
   - GitLab、Gitea 等替代方案尚未形成有效分散

4. **Microsoft 收购后的变化**
   - 社区感知的服务质量下降
   - Copilot 等新产品似乎分散了工程资源
   - 商业化压力 vs 基础设施投入的平衡问题

## 💡 关键洞察 / 为什么重要

### 🏗️ **"Too Big to Fail" 的基础设施困境**

GitHub 已经成为软件世界的**隐性行业标准**：

```
GitHub 故障的影响范围:
├── 所有开源项目的 Issue/PR 管理
├── npm/PyPI/Crates 等包仓库的身份认证
├── GitHub Actions（CI/CD 标准）
├── GitHub Pages（静态托管）
├── Gist（代码片段分享）
└── Copilot（AI 编程助手）
```

**当你的免费工具变成全行业的必需品时，"三个九"就不够好了。**

### 🔄 **去中心化的呼声再起**

这次事件重新点燃了关于**代码托管去中心化**的讨论：

| 方案 | 类型 | 优势 | 劣势 |
|------|------|------|------|
| **GitLab** | 自托管/云 | 功能完整、可自控 | 资源消耗大 |
| **Gitea/Forgejo** | 轻量自托管 | 极轻量、Go 写的 | 功能相对少 |
| **Codeberg** | 社区托管 | 欧洲隐私友好 | 规模小 |
| **SourceHut** | 极简主义 | 无 JS、邮件驱动 | 学习曲线陡 |

### 📈 **可用性的真正成本**

维持高可用性的代价：
- **多区域冗余部署**
- **故障转移自动化**
- **24/7 SRE 团队**
- **混沌工程实践**

问题是：**GitHub 的免费用户是否应该期望企业级可用性？**
- 支持：既然是全球基础设施，就应该按基础设施标准要求
- 反对：免费服务，你得到了你付费的
