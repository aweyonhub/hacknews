## 🎯 核心内容/论点

这是一个关于**开源项目生命周期、代码肥胖症治理和技术债务偿还**的绝佳故事。Video.js 的原始作者 Steve Heffernan 在离开项目 **16 年后重新接管**，带领团队完成了从底层开始的彻底重写。

### 📜 项目背景

**Video.js** 是 Web 上最受欢迎的开源 HTML5 视频播放器之一：
- 创建于 **2010 年**，是最早的 HTML5 视频播放器框架
- 被 Netflix、Instagram、Vox Media 等大量网站使用
- 在 npm 上周下载量达数百万次
- 经过 16 年的发展，积累了大量的功能和……技术债务

### 🔥 核心问题：代码肥胖

重写前的 Video.js 存在严重的问题：

| 指标 | 重写前 | 重写后 | 变化 |
|------|--------|--------|------|
| **包大小** | ~250 KB (gzip) | ~30 KB (gzip) | **↓ 88%** |
| **依赖数量** | 数十个 | 极少 | **大幅减少** |
| **API 复杂度** | 庞大的公共 API | 精简的核心 + 插件 | **模块化** |
| **构建产物** | 多个 bundle | 单一精简 bundle | **简化** |

### 🛠️ 重写策略

#### 1. 基于 Media Chrome 重建
- 采用 **[Media Chrome](https://media-chrome.org)** 作为新的基础架构
- Media Chrome 是一套轻量级的 Web Components 媒体控件标准
- 让 Video.js 从"自己造轮子"转向"站在标准之上"

#### 2. 核心思想转变
```
旧 Video.js: "我们提供一切"
新 Video.js: "我们提供最小核心 + 生态系统"
```

- 核心只保留播放器的基本功能
- 高级功能通过插件机制提供
- 与 React Player 等生态伙伴协作而非竞争

#### 3. Mux 公司的支持
- **[Mux](https://mux.com)**（视频托管/分析平台）深度参与
- Mux 有强烈的动机维护健康的 OSS 视频生态
- Mux 工程师 @luwes 主导了重写工作

## 💡 关键洞察 / 为什么重要

### 📉 **"软件熵增定律"的完美案例**

任何长期维护的开源项目都面临这个困境：

```
第 1 年:  "加这个 feature 很简单，就几行代码"
第 5 年:  "我们需要向后兼容，所以..."
第 10 年: "没人敢删这段代码，不知道谁在用它"
第 16 年: "整个项目就是个胖子，88% 都是历史包袱"
```

Video.js 的故事告诉我们：
> **定期重写不是浪费，而是必要的卫生习惯**

### 🎯 **"回归创始人"模式的启示**

Steve Heffernan 16 年后回归的模式很有趣：
- 他拥有对项目愿景的最原始理解
- 外部视角能发现内部人视而不见的问题
- 但也需要新的团队带来新鲜血液（Mux 的工程师）

这与以下案例类似：
- **Linus Torvalds** 始终在 Linux 核心
- **Guido van Rossum** 回归 Python（BDFL 退休后又回来了）
- **DHH** 始终在 Rails 核心

### 💡 **88% 缩水的具体含义**

减少的不只是字节数，更是：
- **认知负担**：新贡献者更容易理解代码
- **攻击面**：更少的代码 = 更少的安全漏洞
- **加载速度**：对视频首播时间有直接影响
- **维护成本**：更少的 bug 和更快的修复周期
